# U4 — Delete dead config resolve_upstream_chat_kwargs methods + retarget 8 precedence tests

> **Source:** thermo-nuclear PROJECT review 2026-06-20 (Topic 12). See /tmp/thermo-project-review.md

**Priority:** MEDIUM · **Surface:** `src/config.rs`, `tests/port_config.rs` · **Thermo finding:** `Config::resolve_upstream_chat_kwargs` + `resolve_upstream_chat_kwargs_for_resolved_model` (config.rs:833-848) are DEAD in production (post-T1 the leaf finalizes via `BackendFinalizationPolicies::resolve_chat_kwargs`, upstream.rs:1858) yet 8 config tests assert their merge precedence — a path the gateway no longer runs (false confidence, hidden from dead-code lint by `pub`). `resolve_template_family` (config.rs:940) is likewise test-only.

## Purpose
After T1 moved profile finalization to the upstream leaf, the engine no longer asks config to merge `upstream_chat_kwargs`: dispatch builds `BackendFinalizationPolicies::from_config(&config)` once and the leaf calls `resolve_chat_kwargs(&request.model)` (`src/upstream.rs:1858`, invoked from `finalize_request_for_backend` `src/upstream.rs:2011`). The two config methods `resolve_upstream_chat_kwargs` (`src/config.rs:833`) and `resolve_upstream_chat_kwargs_for_resolved_model` (`src/config.rs:838`) have NO production caller (`rg` over `src/` outside `config.rs` finds zero); their only callers are 8 config unit tests (`src/config.rs:2085, 2220, 2291, 2380, 2466, 2606, 2660, 2851`). Worse, those methods compute *different* semantics than the leaf: `model_profiles_for_resolved_model` (`src/config.rs:987`) applies the `upstream_model` remap and collects up to THREE profiles `[resolved_model, configured_model, request_model]`, then `merge_json_maps`-MERGES them in order (`src/config.rs:843-847`), whereas the real leaf's `resolve_chat_kwargs` layers AT MOST ONE per-model policy (`policy_for_model`, exact-then-canonical-unique, `src/upstream.rs:1873`) over the global base (`src/upstream.rs:1859-1862`). So `request_model_profile_overrides_upstream_model_profile_kwargs` (`src/config.rs:2308`) asserts a 2-profile merge `{enabled:true, effort:high}` the gateway never produces — false confidence on a precedence the leaf does not run. `resolve_template_family` (`src/config.rs:940`) is the same: zero production callers, only `src/config.rs:2120/2125/2140` and `tests/port_config.rs:321/326`; the live family resolution is `BackendFinalizationPolicies::resolve_family_override` (`src/upstream.rs:1849`). Deleting the three dead methods and retargeting their tests onto the real leaf restores the tests' truth value. The shared `model_profiles_for_resolved_model` MUST stay — it is still live via `resolve_system_prompt_prefix_for_resolved_model` (`src/config.rs:970`), called from `engine.rs:864`.

## Jobs to Be Done
- The dead `pub` config kwargs/family resolvers are deleted so no test can assert a precedence the gateway no longer runs.
- The 8 kwargs-precedence tests and the family tests are retargeted onto the REAL leaf (`BackendFinalizationPolicies::from_config(&config)` → `finalize_request_for_backend` / its private resolvers), so they validate the path production actually executes.
- The live `model_profiles_for_resolved_model` (system-prompt-prefix) accessor and `resolve_system_prompt_prefix*` are untouched and still pass.
